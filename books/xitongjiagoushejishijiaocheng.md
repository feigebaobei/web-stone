### 系统架构概述

- 系统架构师是项目的总设计师，他需要掌控整体、洞悉局部瓶颈。依据具体的业务场景给出解决方案的总体设计人员。
- 他要确认和主体系统需求，给出开发规范，搭建系统实现的核心架构，并澄清技术细节、扫清主要难点的技术人员
- 掌握技术团队的能力需要，给出项目管理方法，采用合适生命周期模型，具备以自身为核心形成团队的能力。
- 在项目进度计划和经费分配等方面开展评估，以预防项目风险。

系统架构设计的目的对需要开发的系统进行一系列相关的抽象，用于指导系统各个方面的设计与实现。
架构设计的作用主要包括：

- 解决相对复杂的需求分析问题
- 解决非功能属性在系统占据重要位置的设计问题
- 解决生命周期行、扩展性需求高的系统整体结构问题
- 解决系统基于组件需要的集成问题
- 解决业务流程再造难的问题
  软件架构（也可称为体系结构）是用来刻画软件系统整体抽象结构的一种手段。

发展历程

1. 基础研究阶段 1968-1994
2. 概念体系和核心技术形成阶段 1999-2000
3. 理论体系完善与发展阶段 1996-今
4. 普及应用阶段 2000-今

软件架构的常用分类

- 分层架构
- 事件驱动架构
- 微核架构
- 微服务架构
- 云架构

#### 分层架构

最常见的软件架构

- 表现层 用记界面，负责视觉和用户互动
- 业务层 实现业务逻辑
- 持久层 提供数据、sql 语句就在这层
- 数据库 保存数据

```
|--------------------------------------|
| 表现层         组件    组件    组件     |
|--------------------------------------|
|--------------------------------------|
| 业务层         组件    组件    组件     |
|--------------------------------------|
|--------------------------------------|
| 持久层         组件    组件    组件     |
|--------------------------------------|
|--------------------------------------|
| 数据库         数据    数据    数据     |
|--------------------------------------|
```

#### 微核架构

又称为插件架构。指软件的内核相对较小，主要功能和业务逻辑都通过插件实现。
插件是相互独立的。插件之间的通信应该减少到最低，避免出现互相依赖的问题。

```
    即插组件 --- |---------| --- 即插组件
                |         |
    即插组件 --- | 内核系统 | --- 即插组件
                |         |
    即插组件 --- |---------| --- 即插组件
```

#### 微服务架构

是服务导向架构（service-oriented architecture SOA）的升级。
各服务中独立的部署单元，这些单元是分布式的，互相解耦，通过远程通信协议联系。

```
    客户需求    客户需求    客户需求
        ^        ^        ^
        |        |        |
        V        V        V
    |----------------------------------------|
    |            用户接口层                    |
    |----------------------------------------|
        ^        ^        ^
        |        |        |
        V        V        V
    服务组件    服务组件    服务组件
```

3 种实现模式

- RESTful API 模式 服务通过 api 提供，云服务就属于这一类。
- RESTful 应用模式 通过传统的网络协议或者应用协议提供，背后通常是一个多功能的应用程序。
- 集中消息模式 采用消息代码可以实现消息队列、负载均衡、统一日志和异常处理。缺点是会出现单点失败，消息代理可能要做成集群。

#### 云架构

主要解决扩展性和并发的问题。是最容易扩展的架构。

```
    处理单元    处理单元    处理单元
    ^          ^         ^
    |          |         |
    V          V         V
|------------------------------------------------|
|     虚拟中间件                                   |
|  |----------|    |----------|                  |
|  | 消息中间件 |    | 数据中间件 |                 |
|  |----------|    |----------|                  |
|  |----------|    |----------|                  |
|  | 处理中间件 |    | 部署中间件 |                 |
|  |----------|    |----------|                  |
|------------------------------------------------|
```

处理单元 实现业务逻辑
虚拟中间件 负责通信、保持会话控制（sesssion）、数据复制、分布式处理和处理单元的部署
消息中间件 管理用户请求和会话控制，当一个请求进来以后，它决定分配给哪个处理单元。
数据中间件 用于数据同步。使每个处理单元得到同样数据。
处理中间件 可选，当一个请求涉及不同类型的处理单元，则该中间件负责协调处理单元。
部署中间件 负责处理单元的启动和关闭，监控负载和响应时间 ，当负载增加时，则启动新处理单元，当负责减小时，则关闭处理单元。

系统架构的常用建模方法

> 软件架构建模是软件架构的表示方法。

- 结构模型 最直观，最普遍。以架构的构件、连接件和其他概念来刻画结构。
- 框架模型 与结构模型类似，不太侧重描述结构的细节，更侧重整体的结构。建立只针对问题的结构。
- 动态模型 对结构或框架模型的补充。
- 过程模型 研究构造系统的步骤和过程，其结构是遵循过程脚本的结果。

软件架构可从不同角度来描述用户所关心架构的特征。

- 逻辑视角
- 过程视角
- 物理视角
- 开发视角
- 场景视角

不同的架构风格具有不同的优缺点和应用场景
软件架构的应用场景。

- 管道-过滤器风格 适用于系统分成若干独立的步骤
- 主程序、子系统和面向对象的架构风格 用于对组件内部进行设计
- 虚拟机风格 常用于构造解释器或专家系统
- cs/bs 风格 适用于数据和处理分布在一定范围，通过网络连接构成系统
- 平台、插件风格 用于具有插件扩展功能的应用程序
- mvc 风格 广泛应用于用户交互程序的设计
- soa 风格 应用在企业集成等方面
- c2 风格 适用于 gui 软件开发，用于构建灵活和可扩展的应用系统。
  在现代大型软件中，常使用多种架构风格。并保证软件系统的可靠性、可扩展性、可维护性。

#### 存储器

利用半导体、磁、光等介质制成存储数据的电子设备。
可分为 4 个层次

- 片上缓存 处理器核心内的缓存 一般为 SRAM 结构 16k-512k 可分为一级、二级
- 片外缓存 处理器核心外的缓存，需要经过交换互联开关访问，一般是 SRAM 构成。 256k-4m
- 主存（内存） 通常采用 DRAM 结构。以独立的部件、芯片存在。依赖不断充电维持其中的数据。x00m-x0g
- 外存 可以是磁带、磁盘、光盘，访问速度慢，掉电后能够保持其数据。xg-xt。

#### 操作系统

操作系统是配置在计算机硬件上的第一层软件。它向下管理裸机及其文件，向上为其他系统软件提供支持。
操作系统的组成

- 内核
  - 进程管理（任务管理）
  - 存储管理
  - 文件管理
  - 设备管理
- 附加的配套软件
  操作系统的作用
- 管理计算机中运行的程序和分配各种软硬件资源。
  - 资源管理主要包括处理器管理、存储管理、文件管理、i/o 设备管理
- 为用户提供友善的人机界面。
- 为应用程序的开发和运行提供一个高效率的平台。
  操作系统的特征
- 并发性
  - 在一个 cpu 上多次切换程序或多个 cpu 上分别运行一个程序
- 共享性
  - 同时共享
  - 互斥共享
- 虚拟性 指操作系统中的一种管理技术，它是把物理上的一个实体变成逻辑上的多个对应物，或把物理上的多个实体变友逻辑上的一个对应物的技术。
- 不确定性 系统中的进程何时执行、何时暂停，以什么样的速度向前推进、进程总共要花多少时间执行才能完成，这些都是不可预知的。
  操作系统的分类：
- 批处理操作系统
  - 单道批处理
  - 多道批处理
- 分时操作系统
  - 多路性
  - 独立性
  - 交互性
  - 及时性
- 实时操作系统
  - 对交互能力要求不高，对可靠性有保障。
- 网络操作系统
- 分布式操作系统 系统中的计算机无主、次这分。任意 2 台计算机可以通过通信交换信息。
- 微型计算机操作系统
- 嵌入式操作系统
  - 运行在嵌入式智能设备环境中。
  - 微型化
  - 可定制
  - 实时性
  - 可靠性
  - 易移植性

#### 软件构件

构件也叫组件。是一个自包容、可复用的程序集。
软件构件的组装模型的优点：

- 构件的自包含性让系统的扩展变得更加容易。
- 设计良好的构件更容易被重用，降低软件开发成本。
- 构件的粒度较整个系统更小。因此安排开发任务更加灵活。可以把项目会成若干组可以独立开发的组件。
  软件构件的组装模型的缺点：
- 对构件设计需要经验丰富的架构设计师，设计不良的构件难以实现构件的优点，反而会降低组件模型的重用度。
- 需要在重用度与性能之间平衡。
- 要求程序员能够熟练地使用构件组装应用程序。
- 第三方的构件库会影响软件的质量。

### 系统工程

是一种组织管理技术。

- 把研究的对象或工程管理问题看作一个由很多相互联系、相互制约的组成部分构成的总体。
- 运用运筹学的理论和方法以及电子计算机技术，对构成系统的各组成部分进行分析、预测和评价
- 最后进行综合，使该系统达到最优。
  系统工程是运用系统方法，对系统进行规划、研究、设计、制造、试验和使用组织管理技术。
  系统之系统（System of System SoS）适用于其系统元素本身是系统的情况。
  系统工程是从整体出发，合理开发、设计、实施和运用系统科学的工程技术。它根据总体协调的需要，综合应用自然科学和社会科学中有关的思想、理论和方法，利用电子计算机作为工具，对系统的结构、要素，信息和反馈等进行分析，以达到最优规划、最估设计、最优管理和最优控制的目的。
  系统工程专注于：
- 定义客户需求与所要求的功能。将需求文件化。
- 再考试完整问题，即运行、成本、进度、性能、培训、保障、实验、制造、和退出问题，并进行设计综合和系统确认。
  系统工程方法的主要步骤：
- 对系统提出要求
- 根据要求设计系统，评价设计方案。
- 修改要求，再设计。

#### 系统工程的生命周期

定义系统工程的生命周期的目的是以有序而且高效的方式建立一个满足利益攸关者需求的框架。一般通过定义生命周期阶段，并使用一些决策来确定是否处于就绪状态，以便从一个阶段进行下一个阶段来实现这一目的。跳过某些阶段和省去一些决策可能会大幅度增加风险，减少系统工程的投入程度也可能对技术开始造成不利影响。
生命周期阶段

- 探索性研究阶段 目的是识别利益攸关者的需求，探索创意和技术。
- 概念阶段 搜索可行概念，提出有望实现的解决方案
- 开发阶段 旷日持久是细化系统需求，创建解决方案的描述，构建系统，验证并确认系统。
- 生产阶段 系统被生产或制造的阶段
- 使用阶段 是系统在预期环境中运行以交付预期服务的阶段
- 保障阶段 为系统提供服务，使之能持续运行的阶段
- 退役阶段 系统及其相关服务从运行中移除的阶段
  生命周期方法
- 计划驱动方法
- 渐进迭代式开发
  - 允许为项目提供一个初始能力，随之提供连续交付以达到期望的系统。
  - 目标在于快速产生价值并提供快速响应能力。
- 精益开发
  - 是一种整体性的范式，聚集于向客户交付最大价值并使浪费活动最小化
- 敏捷开发
  - 目标在于灵活性
  - 原则
    - 最高的优先级是通过尽早地和持续地交付有价值的软件来满足客户
    - 欢迎需求变更。即使在项目开发后期。
    - 不断交付可用的软件。周期越知越好
    - 在项目中开发人员与业务人员每天在一起工作，业务人员始终参与到开发工作中。
    - 开发团队内，多以面对面式交谈。
    - 工作软件是进度的主要度量
    - 对技术的精益求精以及对设计的不断完善将提升敏捷性
    - 简单性是精髓
    - 最佳的架构、需求和设计出自于自组织的团队
    - 团队要定期反省如何做到更加高效，并相应地调整团队的行为。

## 第 3 章 信息系统基础知识

### 信息系统概述

信息系统是由计算机硬件、网络和通信设备、计算机软件、信息资源、信息用户和规章制度组成的以处理信息流为目的的人机一体化系统。
5 个基本功能：

- 输入功能
- 存储功能
- 处理功能
- 输出功能
- 控制功能
  人类社会的发展速度将取决于人们对信息的利用水平。
  信息系统是以复读机为基础的人机交互系统。
  “以计算机为基础”要求系统设计者必须具备计算机及其在信息处理中的应用知识。
  “人机交互”要求系统设计者还需要了解人作为系统组成部分的能力以及从作为信息使用者的各种行为。

#### 信息系统的发展

数据处理的发展涉及技术的进步、应用拓展、计划和控制策略的变化以及用户的状况。

- 初始阶段
  - 组织中只有个别人具有使用计算机的能力
  - 一般发生在一个组织的财务部门
- 传播阶段
  - 数据处理能力得到迅速发展
  - 出现许多新问题（如数据冗余，数据不一致、难以共享等）
  - 计算机使用效率不高
- 控制阶段
  - 成立了一个领导小组
  - 采用了数据库（db）技术
  - 是计算机管理变为数据管理的关键
- 集成阶段
  - 企业的 it 建设开始由分散和单点发展到成体系，企业 it 主管开始把企业内部不同的 it 机构和系统统一到一个系统中进行管理，使人、财、物等资源信息能够在企业集成共享。
  - 建立集中式的 db 及相应的 is.
  - 增加大量硬件，预算费用迅速增长。
- 数据管理阶段
  - 企业高层意识到信息战略的重要，信息成为企业的重要资源，企业的信息化建设也真正进入到数据处理阶段。在这一阶段中，企业开始选定统一的数据库平台、数据管理体系和信息管理平台，统一数据的管理和使用，各部门、各系统基本实现资源整合和信息共享。it 系统的规划及资源利用更加高效。
- 成熟阶段
  - 从简单的事务处理到支持高效管理的决策企业真正把 it 与管理过程结合起来，将组织内部、外部的资源充分整合和利用，从而提升了企业的竞争力和发展潜力。

### 业务处理系统 Transaction Processing System TPS

tps 主要目的是帮助作业层管理人员减轻处理原始数据的负担，提高具体事务的处理效率。甚至可以完全取代业务层的手工操作。主要放置于车间或一般的行政管理办公室。

### 决策支持系统 Decision Support System DSS

组成：

  定义一：由一个语言系统、知识系统和问题处理系统组成
  特征：
- 数据与模型是 DSS 的主要资源
- DSS 用来支援用户作决策而不是代替用户作决策
- DSS 主要用于解决半结构化及非结构化问题
- DSS 的作用在于提高决策的有效性，不是提高决策的效率
  定义二：
  特征：
- 主要针对上层管理人员经常面临的结构化程序不高、说明不够充分的问题。
- 界面友好，容易被非计算机人员所接受
- 将模型、分析技术与传统的数据存取与检索技术结合起来
- 具有对环境及决策方法改变的灵活性和适应性
- 支持但不是代替高层决策者进行决策
- 充分利用信息技术快速传播和处理信息

  结构：

```

                ^
                |-----------------|
                V                 V

```

功能：总体功能是支持各种层次的人们进行决策

- 用来整理和提供本系统与决策问题有关的各种数据
- 尽可能地收集、存储和及时提供与决策有关的外部信息。
- 及时收集和提供有关各项活动的反馈信息
- 对各种与决策有关的模型具有存储和管理的能力
- 提供常用的数学方法、统计方法、运筹方法、存储、管理
- 支持对数据、模型、方法进行有效管理。为用户提供 curd 操作。
- 用模型、方法对数据加工。
- 具有人机对话接口和图形加工、输出功能。
- 支持分布使用方法，提供有效的传输功能 。

- 在一种网格环境下，提供对数据、模型和方法进行管理功能。
- 具有良好的人机界面的完整软件系统。
- 能对决策起辅助作用的丰富的数据、成熟的模型、有效的方法。

- 面向用户（决策者）
- 支持对半结构化问题的决策
- 辅助决策者、支持决策者。整个决策过程中系统不可能也不应该提供答案，也不应该强加给决策者预告规定的决策顺序。
- 体现决策过程的动态性
- 支持系统提供交互处理

- 数据的重组和确认
  - 事务处理系统的数据必须经过重组和确认才能对决策支持有效。

  - 大多数现存作业层的数据库没有按有利于分析类型和查询应用的方式组织。

  - 再载入关系表
  - 最后数据与逻辑数据模型相一致

  - 运用工具进行数据存取和查询，这个过程是数据挖掘，使用的工具称为智能体。
  - 智能体是管理者用来在关系数据库中搜寻相应数据的软件，用来做趋势分析、异常情况识别和结构跟踪。

    - 定序
    - 分类
    - 聚类
  - 数据仓库的优点：向管理者提供所需要的数据和用来分析这些数据的工具。

  - 目的是帮助决策者理解与选择有关的现象。
  - 大量业务问题需要分析可选的方案设计，模型是建立分析框架的一种有利的工具。
  - 各模型有各自的应用范围
  - 建立一个决策支持系统的难点在于必须清楚系统应包括什么样的模型，如何使这些模型对决策者有意义。

    - 经验型

#### 专家系统的组成

- 知识库
  - 事实性知识
  - 启发性知识
- 综合数据库（又称动态知识库）
  - 是专家系统在执行与推理过程中用以存放所需要和产生的各种信息的工作存储器
- 推理机（又称控制结构或规则解释器）
  - 构成专家系统的核心之一
  - 分类：
    - 正向推理或前向推理 数据驱动策略 适用于目标解空间很大的问题
    - 反向推理或逆向推理 适用于目标解空间较小的问题
    - 双向推理或混合推理
- 知识获取
  - 对知识编辑或求精
  - 自学习
  - 自动修改经验
  - 补充知识库
- 解释程序
  - 面向用户服务的。负责解答用户的问题。
- 人机接口
  - 2 部分
    - 专家系统与用户的接口
    - 专家系统与领域专家户的接口
  - 求解步骤
    - 根据用户的问题在知识库中搜索有相关知识
    - 根据有关的知识和系统的控制策略形成解决问题的途径，即知识操作算子序列。（这时构成一个假设集合）
    - 对解决问题的假设集合进行排序。选了某些准则下的最优假设方案。
    - 根据挑选的解决方案去求解具体问题
    - 如果该方案不能真正解决问题，则追溯到假设方案序列中的下一个，重复求解问题。
    - 重复上述步骤，直到解决问题或宣告“本系统对问题无解”为止。

## 第 4 章 信息安全技术基础知识

#### 信息安全基础知识

信息安全的 5 个要素：

- 机密性 确保信息不暴露给未授权的实体或进程
- 完整性 只有得到允许的人才能修改数据，并且能够判别出是否已被篡改
- 可用性 得到授权的实体在需要时可访问数据。
- 可控性 可以控制授权范围内的信息流向及行为方式
- 可审查性 对出现的信息安全问题提供调查的依据和手段
  信息安全的范围：
- 设备安全
  - 它是首要问题
  - 设备的稳定性 在一定时间内不出故障的概率
  - 设备的可靠性 在一定时间内正常执行任务的概率
  - 设备的可用性 可以正常使用的概率
- 数据安全 即采取措施确保数据免受未授权的泄露、篡改、毁坏
  - 数据的秘密性 不受未授权者知晓的属性
  - 数据的完整性 数据是正确的、真实的、未被篡改的、完整无缺的属性
  - 数据的可用性 数据可随时正常使用的属性
- 内容安全 是信息在政治、法律、道德层次上的要求
  - 信息内容在政治上是健康的
  - 信息内容符合国家的法律法规
  - 信息内容符合中华民族优良的道德规范

### 密钥管理技术

#### 对称密钥的分配与管理

密钥分配一般要解决 2 个问题：

- 引进自动分配密钥机制，以提高系统的效率。
- 尽可能减少系统中驻留的密钥量。
  控制密钥的安全性：
- 密钥标签
- 控制失量
  4 种密钥的分配方式
- 经过 A 选取的密钥通过物理手段发送给另一方 B
- 由第三方选取密钥，再通过物理手段分别发送给 A、B
- AB 事先已经有一个密钥，其中一方选择新密钥后，用已有密钥加密新密钥后发给另一方。
- 三方 ABC 各有一个保密信道。C 选取密钥后，分别通过 AB 各自的保密信道发送。
  no.4 方法最长用。
  会话密钥更换得越频繁，系统的安全性就越高。越频繁更换会话密钥，网络负担越重。需要平衡二者。

#### 公钥加密体制的密钥管理

- 公开发布
  - 用户将自己的公钥发给其他用户。
  - 缺点：任何人都可以发布伪造的别人的公钥。
- 公用目录表
  - 用一个公用的公钥动态目录表，由某个可信的实体或组织去建立、维护公钥的分布。
  - 每个用户与管理者通信，注册、更新自己的公钥。
  - 缺点：若公用目录表被攻击，则有可能被假冒。
- 公钥管理机构
  - 用公钥管理机构来为各用户建立、维护动态的公钥目录。
  - 每个用户都可靠地知道管理机构的公钥，但是只有管理机构知道相应的私钥。
  - 缺点：每个用户想和他人联系时须求助于管理机构。所以很容易使管理机构成为系统的瓶颈。
  - 缺点：若公用目录表被攻击，则有可能被假冒。
- 公钥证书
  - 由证书管理机构 CA（Certificate Authority）为用户建立的，其中的数据项有与该用户的密钥相匹配的公钥、用户身份、时间戳等。
  - 用户将自己的公钥发给另一个用户，则接收方使用 ca 的公钥对证书验证。
  - 这样通过证书交换用户之间的公钥而无须再与公钥管理机构联系。

#### 公钥加密分配单钥密码体制的密钥

```
  |-----|                |-----|
  |     |      1         |     |
  |     | -------------> |     |
  |  A  |      2         |  B  |
  |     | <------------- |     |

1. A用B的公钥加密得到身份IDa和一个用于唯一标记的随机数N1发给B
```

### 访问控制及数字签名技术

#### 访问控制技术

访问控制的基本模型
访问控制是指文体依据某些控制策略或权限对客体本身或是其资源进行的不同授权访问。
访问控制的 3 个要素：

- 主体 对其他实体施加动作的主动实体 简记为 S，有时必称为用户 User，记为 U
- 客体 接受其他实体访问的被动实体 Object 简记为 O
- 控制策略 是主体对客体的行为集和约束条件集 简记为 KS
  访问控制包括：
- 认证
- 控制策略实现
- 审计
  访问控制的实现技术
  建立访问控制模型和实现访问控制都是抽象和复杂的行为。
- 要保证授权用户使用的权限与其拥有的权限对应。
- 制止非授权用户和非授权行为
- 保证敏感信息的交叉感染。
  具体实现方式
- 访问控制矩阵 Access Control Matrix ACM
  - 通过矩阵形式表示访问控制规则和授权用户权限的方法。
  - 对每个主体而言，都拥有哪些客体的哪些访问权限。
  - 对客体而言，有哪些主体可以对它实施访问。
  - 主体是行索引，客体是列索引。矩阵中每个元素表示一组访问方式。
  - 易于理解，实现起来有一定难度。会呈几何数增长。会有好多空值。
- 访问控制表 Access Control Lists ACLs
  - 是目前最流行、使用最多的访问控制实现技术
  - 每个客体有一个访问控制表，是系统中每个有权访问这个客体的主体的信息。
- 能力表 Capabilities
  - 每个主体有一个能力表（Cap-ability Lists），表示主体对系统中每个客体的访问权限信息。
- 授权关系表
  - 是某一个主体对应于某一个客体的访问权限信息
  - 若授权关系表按主体排序，则可以得到能力表的效率；若授权关系表按客体排序，则可以得到访问控制表的效率，

#### 数字签名

向通信双方提供服务，
可以做到：

- B 可以验证消息 P 确实来源于 A
- A 以后不能否认发送过 P
- B 不能编造或改变消息 P
  数字签名的条件
- 签名是可信的。
- 签名不可伪造
- 签名不可重用
- 签名的文件不可改变
- 签名是不可抵赖的。
  对称密钥签名
  公开密钥签名
- 数字签名协议经常和单向 hash 函数一起和，即只对文件的 hash 值签名。
- 优点：
  - 签名和文件可以分开保存
  - 接收者对文件和签名的存储量要求低。
  - 档案系统可用这些协议来验证文件的存在而不需保存它们的内容
  - 用户将文件的 hash 值传给数据库

密钥的选择

  - 用随机数作为会话密钥
  - 随机性不可预测性的要求
  - 每个数出现的频率基本相等
    拒绝服务攻击与防御
    拒绝服务攻击 Dos(Denial of Service)是由人为或非人为发起的行动。使主机硬件、软件或者两者同时失去工作能力。使系统不可访问并因此拒绝合法的用户服务要求。
    DDoS 分布式拒绝服务攻击：
- 服务器的缓存区满，不接收新的请求。
- 使用 IP 欺骗，迫使服务器把合法用户的连接复位，影响合法用户的连接。
  传统拒绝服务攻击的分类
- 消耗资源
  - 大量地申请系统资源
  - 不断地向服务程序发请求
  - 可攻击的资源
    - 针对网络连接
    - 消耗磁盘空间
    - 消耗 cpu 资源和内存资源
- 破坏或更改配置信息
  - 服务程序的配置文件
  - 系统、用户的启动文件
- 物理破坏或改变网络部件
- 利用服务程序中的处理错误使服务失效
  分布或拒绝服务攻击 DDoS
  相比 dos，可以使用更多进攻资源，隐蔽性好。
  被 DDoS 攻击的表现：
- 有大量的等待的 tcp 连接
- 大量到达的数据分组（）并不是网站服务连接的一部分。往往指向机器的任意接口。
- 网络中充斥着大量无用的数据包。源地址为假。
- 制造高流量的无用数据造成网络拥塞，使受害主机无法正常与外界通信
- 利用受害主机提供的服务和传输协议上的缺陷，反复发出服务请求，使受害主机无法及时处理所有正常请求。
- 严重时会造成死机。
  防御拒绝服务攻击的方法
- 加强对数据包的特征识别
  - 通过搜索这些特征字符串，可以确定攻击服务器和攻击者的位置。
- 设置防火墙监视本地主机端口失使用情况。
  - 如果外部主机主动向网络内部高标号商品发起连接请求，则系统可能受到侵入。
- 对通信数据量进行统计也可获得有关攻击系统的位置和数量信息。
  - 攻击数据的来源地址会发出正常极限的数据量
- 尽可能地修正已经发现的问题和系统漏洞。

ARP 欺骗原理
局域网的网络流通是根据 MAC 地址传输的，不是根据 IP 地址传输的。
ARP 欺骗防范措施

- 在 win xp 下输入命令`arp -s gate-way-ip gate-way-mac`固化 arp 表。
- 使用 arp 服务器。
- 采用双向绑定的方法解决并且防止 arp 欺骗
- arp 防护软件

##### DNS 欺骗

DNS 欺骗原理
先冒充域名服务器，后把查询 ip 的地址设为攻击者的 ip 地址。
DNS 欺骗防范措施

- 被动监听检测
- 虚假报文探测
- 交叉检查查询
  - 向 DNS 服务器反向查询应答包中返回的 ip 地址所对应的 dns 名字。若不同则受到攻击。

#### 安全风险管理

风险评估的准备过程是组织进行风险评估的基础。

- 确定风险评估的目标

- 获得最高管理者对风险评估策划的批准
  风险评估的基本要素：
- 脆弱性
- 资产
- 威胁
- 风险
- 安全措施
  要素间的关系：
- 业务战略依赖于资产去完成
- 资产的价值越大，风险越大
- 威胁都要利用脆弱性。脆弱性越大，风险越大。
- 资产的重要性和对风险的意识会导出安全需求
- 安全需求要通过安全措施来得以满足
- 安全措施可以抗击威胁，降低风险，减弱安全事件的影响
- 风险不可能也没有必要降为 0，在实施了安全措施后还会有残留下来的风险。

```
                    业务数据
                      | 依赖
          暴露         V     拥有
  脆弱性 -----------> 资产 -----------> 资产价值
    ^ | ^                              |      ^
    | | |------------------------------+---|  |
    | |------------|                   |   |  |
    |              |           增加     |   |  |
    |利用           |  |----------------|   |  |
    |      增加     V  V      导出           |  |
  威胁 -------------> 风险 --------------> 安全需求
   | ^                 | ^ 降低                 |
   | |攻击              | |-------------------| |被满足
   | |-----------------+------------------|  |  |
   |                   |                  |  |  |
   |演变               |残留                |  |  |
   V        可能诱发    V         未控制     |  |  V
  安全事件 <---------- 残余风险 <----------- 安全措施
```

影响主要从以下几个方面来考虑

- 违反了有关法律或规章制度
- 影响了业务执行
- 造成了信誉、声誉损失
- 侵犯了个人隐私
- 造成了人身伤害
- 对法律实施造成了负面影响
- 侵犯了社会公共准则
- 造成了经济损失
- 破坏了业务活动
- 危害了公共安全
  产生安全威胁的主要因素：
- 人为因素
  - 有意
  - 无意
- 环境因素
  - 不可抗因素
  - 其他物理因素
    风险的计算过程：
- 对信息资产进行识别，并对资产赋值。
- 对威胁进行分析，并对威胁发生的可能性赋值
- 识别信息资产的脆弱性，并对弱点的严重程度赋值
- 根据威胁和脆弱性计算安全事件发生的可能性
- 结合信息资产的重要生和发生安全事件的可能性，计算信息资产的风险值

软件过程模型（软件生命周期模型）：需要一定的工作模型对各项任务给予规程结束。

##### 瀑布模型 Waterfall Model

```
需求分析 --|
          V
      系统分析 --|
                V
            程序分析 --|
                      V
                  编码实现 --|
                            V
                        单元测试 --|
                                  V
                              集成测试 --|
                                        V
                                    系统测试 --|
                                              V
                                          运行维护
```

特点：因果关系紧密相连。
缺点：

- 软件需求的完整性、正确性等很难确定。
- 是一个严格串行的过程模型，使得项目负责人相当长时间看不到软件系统。如果有变更，则有大损失。
- 要求第个阶段一次性地完全解决该阶段的工作。

##### 原型化模型 (快速原型) Prototype Model

```
原型开发 --------------|
                      V
                  编码实现 --|
                            V
                        单元测试 --|
                                  V
                              集成测试 --|
                                        V
                                    系统测试 --|
                                              V
                                          运行维护
```

主要有 2 个阶段：

- 原型开发阶段
  - 开发人员根据软件系统的定义，快速开发一个原型。原型要包含目标系统的关键问题、反映大致面貌、全部或部分功能及性能。
  - 有 3 种途径
    - 利用模拟软件系统的人机界面和人机交互方式
    - 真正开发一个原型
    - 找来一个或几个正在运行的类似软件进行比较
- 目标软件开发阶段
  - 征求用户对原型的意见后修改完善，确认软件系统的需求并达到一致的理解，进一步开发实际系统。
  - 注意事项 - 用户对系统的认识模糊不清，无法准确回答目标系统的需求。
    抛弃型原型：需求确认结束后，原型被抛弃不用，重新采用一个完整的瀑布模型进行开发。
    演化型原型：在需求确认结束后，不断补充和完善原型，直到形成一个完整的产品。

基于快速原型的基础上扩展得到。

### 需求工程

软件需求目前并统一的定义，主要包含：

- 用户解决问题或达到目标所需条件或权能。
- 系统或系统部件要满足合同、标准、规范或其他正式规定文档所需具有的条件或权能。
- 一种反映上面 2 条的文档说明。它包括功能性需求及非功能需求。
  软件需求包括 3 个不同的层次：
- 业务需求 反映了组织机构或客户对系统、产品高层次的目标要求
- 用户需求 描述了用户使用产品必须要完成的任务。是用户对该产品的期望
- 功能需求 开发人员必须实现的软件功能，使得用户能完成他们的任务，从而满足业务需求。
  软件系统规模越大，需求分析与定义整个软件开发与维护过程越重要。
  需求工程的几个阶段：
- 需求获取
- 需求分析
- 形成需求规格
- 需求确认与验证
- 需求管理 包括需求文档的追踪管理、变更控制、版本控制等管理性活动

```
  需求管理
    |
    |--------------|--------------|--------------|
    V              V              V              V
  变更控制        版本控制          需求跟踪        需求状态跟踪
  - 建议变更      - 确定需求文档版本 - 定义对其他需求  - 定义需求状态
  - 分析影响      - 确定单个需求文档   的连接链       - 跟踪需求每一个状态
  - 做出决策        版本           - 定义对其他系统
  - 交流                            元素的连接链
  - 合并
  - 测量需求的稳定性
```

需求管理强调的内容如下：

- 控制对需求基线的变动
- 保持项目计划与需求一致
- 控制单个需求和需求文档的版本情况
- 管理需求和联系链，或管理单个需求和其他项目可交付产品之间的依赖关系
- 跟踪基线中的需求状态

```
                                 V
                               变更实现 --> 修改后的需求
```

变更控制过程是一个渠道和过滤器，通过它可以确保采纳最合适的变更，使变更产生的负面影响降到最低。
常见的需求变更策略：

- 所有需求变更必须遵循变更控制过程
- 对于未获得批准的变更，不应该做设计和实现工作
- 变更应该由项目变更控制委员会决定实现哪些变更
- 项目风险承担者应该能够了解变更的内容
- 绝不能从项目配置库中删除或修改变更请求的原始文档
- 每一个集成需求变更必须跟踪到一个经核准的变更请求，以保持水平可追踪性。

##### 变更控制委员分 Change Control Board CCB

是项目所有者权益代表，负责裁定接受哪些变更。
ccb 是决策机构不是作业机构。
ccb 的成员：

- 产品或计划管理部门
- 项目管理部门

- 测试或质量保证部门

- 制作用户文档的部门
- 技术支持部门
- 帮助桌面或用户支持热线部门
- 配置管理部门
  ccb 应该是一个总则。总则应该说明举行会议的频度和事由。过程如下：
- 制定决策
  - ccb 必须到会的人数或做出有效决定必须出席的人数
  - 决策的方法
  - ccb 的主席是否可以否决该集体的决定
  - 应该权衡利弊

##### 结构化设计 Structured Design SD

是一种面向数据流的设计方法。它以 SRS 和 SA 阶段所产生的数据流图和数据字典等文档为基础，是一个自顶向下、逐步求精和模块化的过程。SD 方法的基本思想是将软件设计成由相对独立且具有单一功能的模块组成的结构。分为概要设计和详细设计 2 个阶段。其中概要设计的主要任务是确定软件系统的结构，对系统进行模块划分，确定每个模块的功能、接口和模块之间的调用关系；详细设计的主要任务是为每个模块设计实现的细节。

模块化。一般具有 3 个基本属性：功能、逻辑、状态。外部特性是模块名、参数表、给程序造成的影响。内部特性是完成其功能的程序代码和仅供内部使用的数据。

###### 系统结构图 Structure Chart SC

又称模块结构图。它是软件概要设计阶段的工具，反映系统的功能实现和模块之间的联系与通信，包括名模块之间的层次结构，即反映了系统的总体结构。
详细设计的主要任务是设计每个模块的实现算法、所需的局部数据结构。目标：

- 实现模块功能的算法要逻辑上正确和算法描述要简明易懂。
- 若要更改概要设计，则必须经过项目经理的同意。
  设计的基本步骤：

1. 分析并确定输入、输出数据的逻辑结构
2. 找出输入数据的结构和输出数据结构中对应关系的数据单元
3. 按一定的规则由输入、输出的数据结构导出程序结构。
4. 列出基本操作与条件，并把它们分配到程序结构图的适当位置。
5. 用伪码写出程序。
   详细设计的表示工具：

- 图形工具
  - 业务流图
  - 程序流程图（也叫程序框图）
    - 优点：结构清晰，易于理解，易于修改。
    - 缺点：只能描述执行过程，不能描述有关的数据
  - NS 流程图（也叫盒图）
  - Problem Analysis Diagram PAD 图
    - 可取代程序流程图。
    - 特点：
      - 使用 pad 符号设计出的程序代码是结构化程序代码
      - pad 所描绘的程序结构十分清晰
      - 用 pad 图表现程序的逻辑易读、易懂和易记
      - 容易将 PAD 图转换成高级语言源程序自动完成
      - 既可以表示逻辑，也可用来描绘数据结构。
      - 支持自顶向下方法的使用。
- 表格工具
  - 用一张表来描述过程的细节，这张表中列出了各种可能的操作和相应的条件。
- 语言工具 Program Design Language PDL

本质是主张参照人们认识一个现实系统的方法，完成分析、设计与实现一个软件系统，提倡用人类在现实生活中常用的思维方法来认识和理解描述客观事物，强调最终建立的系统能映射问题域，使得系统中的对象，以及对象之间的关系能够如实地反映问题域中固有的事物及其关系。
面向对象开发方法是

- 以用例驱动的
- 以体系结构为中心的

- 需求分析

##### 面向对象分析 Object-Oriented Analysis OAO

是在一个系统的开发过程中进行了系统业务调查以后，按照面向对象的思想来分析问题。
由 5 个层次：

- 主题层
- 对象类层
- 结构层
- 属性层
- 服务层
  和 5 个活动组成
- 对象类
- 标识结构
- 定义主题
- 定义属性
- 定义服务
  2 种对象类之间的结构：
- 分类结构
- 组装结构
  OOA 原则：
- 抽象
  - 抽象 任何一个完成确定功能的操作序列，其使用者都可以把它看作一个单一的实体，尽管实际上它可能是由一系列更低级的操作完成的。
  - 数据 根据施加于数据之上的操作来定义数据类型，并限定数据的值只能由这些操作来修改和观察。 它是 ooa 的核心原则。
- 封装 把对象的属性和服务结合为一个不可分的系统单位。
- 继承 可使系统模型比较简练也比较清晰。
- 分类
- 聚合
- 关联
- 消息通信 对象之间只能通过消息进行通信。由于封装原则而引起。
- 粒度控制
- 行为分析
  基本步骤

1. 确定对象和类
2. 确定结构
3. 确定主题
4. 确定属性
5. 确定方法

##### 面向对象设计 Object-Oriented Design OOD

是 OOA 方法的延续。
3 种类型：

- 实体类
  - 映射需求中的每个实体。指实体类保存需求存储在永久存储体中的信息。
  - 一定有属性，不一定有操作。
- 控制类
  - 由动宾结构的短语命名。
  - 用于控制用例工作的类。
  - 它的行为具有协调性。
  - 通常控制其他对象。
  - 没有属性，一定有方法。
- 边界类
  - 位于系统与外界的交接处。
  - 有属性，有方法。

### 软件测试

使用人工或自动的手段来运行或测定某个软件系统的过程，其目的在于检验它是否满足规定的需求或弄清预期结果与实际结果之间的差别。
重要的是软件测试要贯穿在整个软件开发的过程中，保证整个软件开发的过程是高质量的。

#### 测试方法

按执行状态分为静态测试、动态测试
按实现算法细节和系统内部结构分为黑盒测试、白盒测试、灰盒测试。
按程序执行方式分为人工测试、自动化测试

- 静态测试
  - 以源程序做结构分析和流程图分析
  - ts 就是使用这种测试
- 动态测试
  - 构造测试实例
  - 执行程序
  - 分析结果
- 黑盒测试
  - 不考虑内部结构，根据需求规格测试实例。检查功能是否正确运行。
  - 主要对软件界面和软件功能进行测试
  - 必须加以量化才能够保证软件的质量。
- 白盒测试
  - 从程序结构方面设计测试用例。用于检查各逻辑是否合理。
  - 对应的模块独立路径是否正常。
  - 常用方法
    - 控制流分析
    - 数据流分析
    - 路径分析
    - 程序变异
- 灰盒测试
- 人工测试
- 自动化测试

#### 测试阶段

- 单元测试
  - 对软件的模块进行测试
- 集成测试
  - 对组装起来的模块进行测试，明确其正确性、发现接口的问题
- 系统测试
  - 功能测试
  - 性能测试
  - 健壮性测试
  - 安装或反安装测试
  - 用户界面测试
  - 压力测试
  - 可靠性及安全性测试
- 性能测试
  - 用自动化测试工具模拟多种正常、峰会、异常负载条件对系统的各项性能指标。
  - 确定一个系统的瓶颈或者不能接受的性能点。获得系统能提供的最大服务级别的测试。
- 验收测试
  - 要满足用户需求或者用户签订的合同
  - 验证实际工作的有效性和可靠性。确保用户能用该软件顺利完成任务和功能。
- 其他测试
  - ab 测试
  - web 测试
  - 链接测试
    - 所有链接的 url 是否正确
    - url 对应的页面是否正确
    - 是否有独立的页面
  - 表单测试

### 净室软件工程 Cleanroom Software Engineering CSE

是一种应用数学与统计学理论以经济的方式生产高质量软件的工程技术。力图通过严格的工程化的软件过程达到开发中的零缺陷或接近零缺陷。
要求在规约和设计中消除错误，然后有“净”的方式制作。
通过在第一次正确地书写代码增量，并在测试前验证它们的正确性，来避免对成本很高的错误消除过程的依赖。
净室是一种以合理的成本开发高质量软件的基于理论、面向工作组的方法。

#### 理念基础

- 函数理论
- 抽样理论

#### 技术手段

- 统计过程控制下的增量式开发 Incremental Development
- 基于函数的规范与设计
- 正确性验证
- 统计测试（Statistically Based Testing）和软件认证

#### 应用与缺点

净室软件工程有真实的价值和质量优势。
缺点：

- 太理论化
- 开发小组不进行传统的模块测试
- 不可避免带有结构化开发的弊端

### 软件项目管理

#### 项目管理概述

软件项目管理和其他的项目管理相比有一定的特殊性。

- 软件是纯知识产品。
- 软件系统的复杂性导致了开发过程中各种风险的难以预见和控制
- 管理工作在技术工作开始前开始，在软件从概念到实现的过程中继续进行，当软件工程过程最后结束时终止。

#### 软件进度管理

按时完成软件项目是项目经理最大的挑战之一。

- 工作分解结构 Work Breakdown Structure WBS
  - wbs 的工作包是可控和可管理的。不能过于复杂。
  - 任务分解也不能过细，一般原则 WBS 的树结构不超过 6 层
  - 每个工作包要有一个交付成果
  - 每个任务必须有明确定义的完成标准
  - WBS 必须有利于责任分配
- 任务活动图
  - 需要明确每个活动的前驱、持续时间、必须完成日期、里程碑、交付成果。
  - 一般采用甘特图管理项目活动

#### 软件配置管理 Software Configuration Management SCM

- 版本控制
  - 追踪文件的变更
  - 重要工作并行开发
- 变更控制
  - 来自外部的变更要求
  - 开发内部的变更要求

#### 软件质量管理

软件质量是软件与明确地、稳含地定义的需求相一致的程序。
可以影响软件质量的因素可分 3 组

## 第 6 章 数据库设计基础知识

### 数据库基本概念

- 数据 data 描述事物的符号记录。有多种表现形式，可以是文字、图形、图像、声音、语言。
- 信息 information 是现实世界事物的存在方式或状态的反映
- 数据库系统 DataBase System DBS 一个采用了数据库技术，有组织地、动态地存储大量相关数据，从而方便多用户访问的计算机系统。
- 数据库 DataBase DB 是统一管理的、长期储存在计算机内的，有组织的相关数据的集合。
- 数据库管理系统 DataBase Management System DBMS 是数据库系统的核心软件，由一组相互关联的数据集合和一组用以访问这些数据的软件组成。

#### 数据库技术的发展

数据处理是对各种数据进行收集、存储、加工、传播的一系统活动。
数据管理是对数据进行分类、组织、编码、存储、检索、维护的活动。

1. 人工管理阶段
   - 特点
     - 数据量小
     - 数据不保存
     - 没有软件系统管理数据
   - 缺点
     - 应用程序与数据之间依赖性强
     - 数据组之间会重复许多数据。
2. 文件系统阶段
   - 按一定规则把数据组织在一个文件中。应用程序通过文件系统对文件中的数据进行存取和加工。
   - 特点
     - 数据可以长期保留
     - 数据不属于某个特定的应用。
     - 文件组织形式的多标准化
   - 缺点
     - 数据冗余
     - 数据不一致性
     - 数据孤立
3. 数据库系统阶段
   - 由计算机软件、硬件资源组成的系统，它有组织地、动态地存储大量关联数据，方便多用户访问。
   - 特点 - 采用复杂的数据模型表示数据结构 - 有较高的数据独立性

#### 数据需求分析

- 面向企业和部门的具体业务

- 自顶向下
- 自底向上

#### 概念结构设计

目标是产生反映系统信息需求的数据库概念结构，即概念结构。
实体-联系方法 Enity-Relationship Approach 将现实世界的信息结构统一由实体属性，以及实体之间的联系来描述。

- 选择局部应用
- 逐一设计分 ER 图
- ER 图合并
  使之成为能够被全系统所有用户共同理解和接受的统一的、精练的僵尸概念模型。
  合并 ER 图时的冲突主要有 3 种：
- 属性冲突
- 命名冲突
- 结构冲突
  合并时可以优化：
- 实体类型合并
- 冗余属性的消除
- 冗余联系的消除

#### 逻辑结构设计

在概念结构设计的基础上进行数据模型设计。

- ER 图转换为关系模式

  - ER 图中的联系有 3 种

    - 一对多联系
    - 多对多联系
- 关系模式规范化
- 确定完整性约束
- 确定用户视图
- 反规范化
  - 是快速读操作性能的方法

#### 物理设计

1. 确定数据分布
   - 目前一般采用分布式管理
   - 需要考虑 3 个方面
     - 根据不同应用分布数据
     - 根据处理要求确定数据的分布
     - 对数据的分布存储必然会导致数据的逻辑结构的变化
2. 确定数据的存储结构
   - 在实际应用中，要根据数据的处理要求和变更频度选定合理的物理结构。
3. 确定数据的访问方式
   - 由其存储结构所决定。
   - 存储记录结构设计
   - 存储记录布局
     - 聚簇码的值相对稳定，没有或很少需要进行修改。
     - 表主要用于查询，并且通过聚簇码进行访问或连接是该表的主要应用。
     - 对应每个取簇码值的平均元组数既不太多，也不太少。
   - 存取方法的设计
     - 存取方法是为存储在物理设备上的数据提供存储和检索的能力，是快速存取数据库中的数据的技术
     - 建立索引的方式
       - 首先可确定关系的存储结构
       - 确定不宜建立索引的属性或表
       - 确定宜建立索引的属性

### 应用程序与数据库的交互

DBMS 应该提供程序级别的接口来访问数据。

#### 库函数级别访问接口

往往是数据库提供的最底层的高级程序语言的访问数据接口。
函数包含：

- 连接数据库
- 调用 SQL
- 事务控制
  此方法的缺点是往往强依赖于特定的数据库，需要数据库开发人员对该数据库机制有较深的理解，学习难度较大，开发效率不是很高。

#### 嵌入 SQL（Embedded SQL）访问接口

是一种将 SQL 语句直接写入某些高级程序语言。
可使得应用程序拥有了访问数据以及处理数据的能力。
嵌入式 SQL 在技术上必须解决的问题：

- 宿主语言的编译器不能识别和接受 SQL 程序，应该解决如何将 SQL 的宿主语言源代码编译成可执行码。
- 宿主语言的应用程序如何与 DMBS 之间传递数据和消息
- 如何把对数据的查询结果逐次赋值为宿主语言程序中的变量，以供其处理。
- 数据库的数据类型与宿主语言的类型有时不完全对应或等价，如何解决必要的数据类型转换问题。

#### 通用数据接口标准

开放数据库连接 Open DataBase Connectivity ODBC 是为解决异构数据库间的数据共享而产生的。
ODBC 为异构数据库访问提供统一接口，允许应用程序以 SQL 为数据存取标准，存取不同 DBMS 管理的数据。
使应用程序直接操纵数据库中的数据，免除随数据库的改变而改变，也可以访问如 Excel 表和 ASCII 数据文件这类非数据库对象。
ODBC 对数据库的操作不依赖任何 DBMS。
优点：以统一的方式处理所有的关系数据库。
具体操作方法：

- 数据库访问对象 Database Access Object DAO
- 远程数据库对象 Remote Database Object RDO
- ActiveX 数据对象 ActiveX Data Objects ADO

#### ORM 访问接口

对象关系映射 Object Relational Mapping ORM 是一种程序设计技术。用于实现面向对象编程语言里不同类型系统数据之间的转换。
ORM 是通过使用描述对象和数据库之间映射的元数据，将程序中的对象与关系数据库相互映射；
ORM 是一种 将内存中的对象保存到关系型数据库中的技术，主要负责实体域对象的持久化。
典型的 ORM 框架：

- Hibernate 全自动框架
- Mybatis 半自动框架
- JPA java 自带的框架

### NoSQL 数据库

Non-Relational Not Only SQL 仅仅是一个概念，泛指非关系型的数据库，区别于关系数据库，它们不保证关系数据的 ACID 特性。

#### 分类与特点

- 列式存储数据库
  - 行式数据库即关系型数据库，数据按记录存储，每一条记录的所有属性存储在一行。
- 键值对存储数据库
  - 用一个哈希表，表中有一个特定的键和一个指针指向特定的数据。
- 文档型数据库
  - 允许嵌套键值，在处理网页等复杂数据时，文档型数据库比关系型数据库的查询效率更高。
- 图数据库
  - 它使用灵活的图形模型，并且能够扩展到多个服务器上。
  - 适合存储通过图进行建模的数据
    NoSQL 的优点：
- 易扩展
- 大数据量
- 灵活的数据模型
- 高可用

#### 体系框架

共 4 个层次：

- 接口层 Interface
  - 5 个选择
    - Rest
    - Thrift
    - Map/Reduce
    - Get/Put
    - 特定语言的 API
- 数据逻辑模型层 Data Logical Model
- 数据分布层 Data Distribution Model
  - 定义数据如何分布。
    - CAP 支持，可用于水平扩展
    - 多数据中心支持，可以保证在横跨多数据中心时平稳运行
    - 动态部署支持，可以在运行着的集群中动态地添加或删除节点
- 数据持久层 Data Persistence
  - 定义了数据的存储形式
    NoSQL 的适用场景：
- 数据模型比较简单
- 需要灵活怀更强的 it 系统
- 对数据库性能要求较高
- 不需要高度的数据一致性

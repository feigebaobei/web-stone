软件架构师手册

前言

本书的目的：帮助软件开发人员过渡为软件架构师。帮助现有架构师进步。 了解架构师的工作内容、职位责任。同时具备技术与非技术的能力；具备知识的广度与深度。

1 软件架构的含义

ISO/IEC/IEEE 42010 标准定义

  • 软件架构是软件系统最基本的组成部分软件系统。
  • 软件系统处于一个特定的环境中，它的架构也要考虑其所运行的环境。
  • 架构描述对架构进行了文档化的记录，并向利益想着者传达了特定的架构是如何满足特定系统的需求。
  • 架构视图是从架构描述中创建出来的。每个视图涵盖了利益相关者的一个或多个关注点。

软件架构只关注重要的东西

由对系统的质量、寿命和有用性至关重要的结构和组件构成。

定义满足需求的解决方案

软件力求满足所有功能性、非功能性、技术性和操作性的需求。与包括领域专家、业务分析师、产品负责人和最终用户在内的利益相关者密切合作，可以让需求被识别和理解。软件架构师则定义满足这些需求的解决方案。

管理变更。

变更可分为以下类别：

  • 局限于单个元素。
  • 涉及一组元素，但不需要架构调整。
  • 需要架构调整。

不要害怕变更，正面变更，做出调整。 有时刚设计好解决方案变更就来了；有时开发一半时变更来了；有时部署后变更来了。

提供可重用模型

当一个软件架构被重用时，从原始架构设计中获得的经验同样可以用于其他软件系统。

实施实现约束

  • 使开发人员完成目标。
  • 预防开发时犯错误。

改进成本和工作评估

软件架构师提供信息（如：完工时间、成本）使项目经理正确规划资源并监控进度。常用方法：

  • 自顶向下：从最终的可交付成果和目标出发，再分解为更小的工作组合。
  • 自底向上：从特定的任务出发。组合到工作组合中。

培训团队成员

软件系统的维护阶段是软件项目中耗时最长，成本最高的阶段。

开发者维护软件，架构师维护开发者。

软件架构不是银弹

软件架构是所有已知问题的折中解决方案。

软件架构师是技术领导者

软件架构师需要履行多种职责

  • 必须对软件架构设计、架构模式以及最佳实践有着深刻的理解。
  • 应当预见可能出现的问题并设计出解决方案。如：降低风险并评估不同的解决方案。选出最合适的解决方案。
  • 对技术有深度有广度。了解使用过、没有使用过的技术。

理想情况下，软件架构师具有雄厚的知识，了解一个问题的解决方案并且能权衡各方案的利弊。对于软件架构师而言，理解一个解决方案为什么不能用与理解其为什么能用同样重要。

软件架构师需要掌握的知识

  • 非技术性
    • 发挥领导作用
    • 协助项目管理，包括成本和工作量评估。
    • 指导团队成员。
    • 协助选择团队成员。
    • 理解业务领域
    • 参与需求的收集和分析
    • 与各技术性、非技术性的干系人沟通。
    • 对未来的产品有清晰的愿景。
  • 技术性
    • 理解非功能性需求以及质量属性。
    • 能够有效地设计软件架构
    • 理解软件开发的模式和最佳实践。
    • 对软件架构模式及其优缺点有着深入理解，知道如何取舍。
    • 了解如何处理横切关注点
    • 确保满足性能与安全性需求。
    • 能够对软件架构进行记录和审查
    • 理解 devops 及其部署流程
    • 了解如何集成并使用遗留程序。
    • 能够设计适应变化和随时间演化的软件架构。

4 软件质量属性

质量属性

  • 是软件系统的非功能性属性。
  • 它应用于可测量、可测试的。
  • 描述软件系统质量和度量系统适用性的基准。
  • 软件系统由质量属性的组合构成，它们被满足的程序反映了软件的整体质量。
  • 有些软件质量属性之间互相影响。

解决方案就是在众多质量属性之间达到平衡。

测试质量属性

  • 手动测试软件的可用性
  • 为性能测试建立基准并使用工具进行测试
  • 执行代码审查和计算代码度量以测试其可维护性
  • 执行自动化单元测试以确保系统按预期运行

软件架构设计

  • 为满足功能需求、质量属性和约束做出决策。
  • 是一个创造性的过程。软件架构师利用自己的创造性为复杂的问题提出解决方案。这可能是软件项目中最有趣和最有价值的部分了。
  • 设计术语：结构、元素、系统、子系统、模块、组件
  • 逃避设计决策可能导致技术债务。
    • 技术债务：由于当时做出的或者没有做出的决策导致后来需要耗费额外的工作成本和精力。
    • 不全是坏事。有时会为了眼前的利益付出一些代价。

自顶向下方法和自底向上方法

  • 自顶向下方法
    • 从整个系统的最高层开始，从上到下进行分解。
    • 优点
      • 是一种设计和分解的系统化方法。
      • 适合分工
    • 缺点
      • 大量预先设计
    • 适用条件
      • 项目规模大、复杂。
      • 是为组织设计的。
      • 一个项目多个合作团队
      • 充分理解领域
  • 自底向上方法
    • 从解决方案所需要的组件开始，逐步向上设计直到较高的抽象级别。
    • 优点
      • 简单。只关注独立的部分，并且一个迭代中只构建需要的部分。
      • 无大量预先设计。
      • 促进代码重用。
    • 缺点
      • 低估了变更的代码。
      • 较差的可维护性。
      • 设计缺陷可能不会在早期被发现。
    • 适用条件
      • 项目规模小、简单
      • 不是为组织设计的
      • 只有一个开发团队
      • 未充分理解

松耦合

耦合：一个软件模块依赖于另一个软件模块的程度。

耦合类型

  • 内容耦合 一个模块直接引用另一个模块的内部或私有信息
  • 公共耦合（全局耦合）
  • 外部耦合 多个模块共享软件的相同部分的外部环境
  • 控制耦合 一个模块通过传递信息的方式来控制另一个模块的内部逻辑。
  • 戳耦合（数据结构耦合） 某种内部结构的数据
  • 数据耦合 2 个模块共享一些原始数据值
  • 消息耦合 一个模块调用另一个模块上的方法而不发送任何参数
  • 无耦合 2 个模块完全没有任何直接通信的情况

松耦合设计

  • 最小化模块之间的耦合量。模块应该尽可能独立。减小不必要的耦合。

11 安全性注意事项

确保软件系统的安全

架构的安全性，必须在需求、设计、开发和测试期间都考虑到。 信息的三种状态：

  • 静息状态，未被访问。如数据库的文件。
  • 使用状态，正在被进程使用的信息。如在内存（或 cpu）中的数据。
  • 传输状态，正在被移动的信息。

三要素

  • 机密性 confidentiality 防止包括未经授权的个人访问消息。应用程序必须保存其数据，特别是当数据是属于个人私有的。
  • 完整性 integrity 防止未经授权的个人修改或破坏信息。
  • 可用性 availability 防止包括未经授权的个人访问消息。

是分析应用程序安全性的一种结构性方法。威胁是指对软件系统可能存在的危险。 分解应用程序：

部署到云端

云的类型

  • 公有云
    • 由第三方云供应商拥有。
    • 大量资源可以让使用它的组织体验到高可靠性和实际上无限的可伸缩性。
    • 组织需要按使用量付费，不必担心资源的维护。
    • 不适合于有敏感或受法律管控的数据。
    • 失去数据的管理权。
  • 私有云
    • 成本高，经济规模小。
    • 也可以由第三方云提供商实现。
    • 组织对流程、数据管理政策和物理资源有绝对的控制权
    • 组织负责私有云的安全
    • 中小型企业较难达到主流云提供商的可靠性、效率、可伸缩性水平。
  • 混合云
    • 高容量、低安全需求的功能托管在公有云。
    • 关键任务和功能和敏感数据托管在私有云。
    • 混合云可成为组织向云逐步过渡的阶段。
    • 可实现云爆发（也称为爆发计算模式 burst compute pattern）。应用程序运行在私有云，当需求激增时进入公有云。

云模型

  • 基础设施即服务 IaaS
    • 供应商维护硬件，其他的由组织负责。
    • 常见诉求：
      • 网站托管
      • web 应用程序
      • 测试、开发环境
      • 存储、备份、恢复需求和大数据分析
    • 常用于在云服务上起步
  • 容器即服务 CaaS
    • 用于云原生软件的开发和部署模型
    • 在 IaaS 的基础上添加了容器编排平台
    • 可以使用开发团队控制应用程序和依赖项的打包方式，从而使应用可以移植到各个环境。
  • 平台即服务 PaaS
    • 供应商为组织提供了开发和部署应用程序的完整平台。如大量的支持软件、工具、服务、数据库管理系统和中间件。
    • 组织失去了对操作系统的控制。
  • 无服务、函数即服务 FaaS
    • 根据每一次执行的模型来收费。
  • 软件即服务 SaaS
    • 也称为按需软件。
    • 组织负责软件的配置。
    • 组织不需要为使用软件支付费用。

不同这处在于它们的抽象级别、控制和责任的数量。 抽象级别递增。

14 遗留应用架构设计

遗留应用程序

仍在使用但难以维护的程序。 遗留应用程序存在的问题：

  • 难以维护和扩展
  • 使用较旧的、过时的技术
  • 有可能没有遵循软件开发的最佳实践过程
  • 软件熵高（多人维护）
  • 有一定量的技术债务

为什么要使用遗留应用程序

  • 系统仍然有用并且按预期工作。（没有理由停掉它们）
  • 替换遗留系统的成本可能会超过收益。
  • 遗留应用程序没有被理解透彻。

遗留应用程序包括：

  • 代码
  • 运行所需要的环境
  • 特定的工具。如代码编辑器、版本控制、构建工具、代码覆盖、代码评审、调试、集成、文档、静态代码分析、单元测试工具。
  • 其依赖组成。如：第三方软件、框架、包。

重构遗留应用程序

当开始处理遗留应用程序时，就应该考虑对其进行重构，使其易于维护。未来迭代中可能需要实现新特性、修复缺陷、优化设计、提高质量、优化应用程序。为了执行这类任务，该程序必须处于一种无风险地、简单地进行修改的状态。

在不改变代码的外部行为又能优化代码的内部结构的修改软件系统的过程。 尊重最初的开发团队。 重构的任务：

  • 让遗留代码可测试
    • 增加单元测试
    • 单测的好处
      • 为系统修改提供便利
      • 确保新修改不会引入新的缺陷。
      • 单测可以成为文档的一个来源。
      • 使团队熟悉代码库。
    • 在哪里开始编写测试
      • 从最关键的业务功能相关的逻辑开始
      • 从最复杂的组件开始（适合有经验的团队）
      • 从最简单的组件开始（适合无经验的团队）
  • 移除冗余代码
    • 不可触达的代码：永远不会执行的代码。
    • 死代码：可以执行，但是不使用执行的结果的代码。
    • 注释掉的代码。
    • 重复的代码。
  • 使用工具重构代码
  • 做出小的、增量的修改。
    • 修改不应过大
    • 重构遗留程序可能会持续很长时间
    • 在当前迭代中修改相关代码
  • 将单体系统转化为微型服务

硬技能是可以定义和度量的具体技能。通常针对具体的工作，可以通过教育、培训以及认证获得。 软技能是无形的技能，难以定义和度量。

16 演进架构

在软件开发方法中，大量时间花在了大量预先设计上。 如何创建一个能够适应变化的软件架构。

变化不可避免

改变的原因：

  • 编辑语言、框架、持久性技术和工具。
  • 组织的业务

期待改变

  • 软件架构师应该预料到变化。
  • 设计一个演进的软件架构能够处理不可避免的变化。

Lehman 的软件演进法则

S 型系统

它是一个众所周知的、精确的规格。可以依据规格进行开发。需求不太会变动。 如：计算器程序、精确数字计算的程序。

P 型系统

可以精确阐述问题的系统。可以被明确知晓，为系统提供一个精确的规格。 如：国际象棋程序。

E 型系统

基于真实世界的流程和人员建模。 大多数软件系统是 E 型系统。 术语“嵌入”指把系统嵌入到现实世界中。 需求不断变化。

法则

软件演进相关的现象和行为

  • 法则一 需求持续变更
  • 法则二 复杂性递增
  • 法则三 自我调节
  • 法则四 组织稳定性
  • 法则五 熟悉度恒定
    • 团队成员必须始终对系统保持相同水平的熟悉程度。
  • 法则六 持续增长
    • 持续发挥作用
    • 满足用户的需求
    • 功能数量增加
  • 法则七 质量衰减
  • 法则八 反馈系统
    • 收集所有利益相关者的反馈

设计演进架构

努力将软件系统设计为高度可维护的易于更改的。演进体系架构的核心思想是能够轻松地更改软件架构。一个演进应该支持不涉及架构变更的修改。

一个演进的架构支持跨多个维度的受引导的增量变更 进行受引导的架构变更。软件架构师应该引导软件系统架构所发生的任何更改，来保证架构的特征保持稳定。软件架构是由设计决策所塑造的。这些设计决策赋予软件特定的属性， 架构必须适配一些技术或业务更改时，不能只是简单地随意一个能支持该适配修改上线。软件系统是动态的。需要充分评估变动的后果。在多次迭代后系统的质量破败在一定范围内。 软件架构师必须对变更有所引导。

适应函数

适应函数是一种引导架构变更方法，可以帮助软件架构师确定修改的影响。

  • 用于确定给定的解决方案和预期结果的差距。
  • 可返回解决方案的合适程序
  • 被用于遗传算法的设计
  • 生成给定问题的最优解

适应函数分类：

  • 原子与整体
    • 所有单一架构通过单元测试
    • 重要特征的组合测试通过
  • 触发和连续
    • 触发适应函数会基于某个事件被执行。
    • 一个连续的适应函数会一直持续执行。
  • 静态与动态
    • 当测试的条件值恒定时，就是静态适应函数。反之为动态适应函数。
  • 自动和手动
  • 时间
    • 基于特定的时间量
  • 有意和浮现
    • 有意适应函数：在项目早期定义了许多适应函数。
    • 涌现函数：在开发中定义的适应函数

设计可发展的 api

将 Postel 法则应用于 api

  • 坚固性原则 Robustness Principle （也称为 Postel 法则）是消息契约设计指导原则。
    • 应该在当前模块上所做的事件上保持保守。在接收其他模块上保持自由。
    • 最初是为了 tcp 设计的。（在发出的东西上保守，在接收的东西上自由。）
    • 最低限度地发送到系统外。
    • 消息发送者必须遵守消息契约，减小 api 暴露的数据量。使造成破坏性的机会减小。
    • 接收时自由：从消息中提取需要的内容，忽略不需要的内容。

系统架构概述

  • 系统架构师是项目的总设计师，他需要掌控整体、洞悉局部瓶颈。依据具体的业务场景给出解决方案的总体设计人员。
  • 他要确认和主体系统需求，给出开发规范，搭建系统实现的核心架构，并澄清技术细节、扫清主要难点的技术人员
  • 掌握技术团队的能力需要，给出项目管理方法，采用合适生命周期模型，具备以自身为核心形成团队的能力。
  • 在项目进度计划和经费分配等方面开展评估，以预防项目风险。

系统架构设计的目的对需要开发的系统进行一系列相关的抽象，用于指导系统各个方面的设计与实现。 架构设计的作用主要包括：

  • 解决相对复杂的需求分析问题
  • 解决非功能属性在系统占据重要位置的设计问题
  • 解决生命周期行、扩展性需求高的系统整体结构问题
  • 解决系统基于组件需要的集成问题
  • 解决业务流程再造难的问题 软件架构（也可称为体系结构）是用来刻画软件系统整体抽象结构的一种手段。

3 种实现模式

  • RESTful API 模式 服务通过 api 提供，云服务就属于这一类。
  • RESTful 应用模式 通过传统的网络协议或者应用协议提供，背后通常是一个多功能的应用程序。

系统架构的常用建模方法

软件架构建模是软件架构的表示方法。

  • 结构模型 最直观，最普遍。以架构的构件、连接件和其他概念来刻画结构。
  • 框架模型 与结构模型类似，不太侧重描述结构的细节，更侧重整体的结构。建立只针对问题的结构。
  • 动态模型 对结构或框架模型的补充。
  • 过程模型 研究构造系统的步骤和过程，其结构是遵循过程脚本的结果。

软件架构可从不同角度来描述用户所关心架构的特征。

  • 逻辑视角
  • 过程视角
  • 物理视角
  • 开发视角
  • 场景视角

不同的架构风格具有不同的优缺点和应用场景 软件架构的应用场景。

  • 管道-过滤器风格 适用于系统分成若干独立的步骤
  • 主程序、子系统和面向对象的架构风格 用于对组件内部进行设计
  • 虚拟机风格 常用于构造解释器或专家系统
  • cs/bs 风格 适用于数据和处理分布在一定范围，通过网络连接构成系统
  • 平台、插件风格 用于具有插件扩展功能的应用程序
  • mvc 风格 广泛应用于用户交互程序的设计
  • soa 风格 应用在企业集成等方面
  • c2 风格 适用于 gui 软件开发，用于构建灵活和可扩展的应用系统。 在现代大型软件中，常使用多种架构风格。并保证软件系统的可靠性、可扩展性、可维护性。

存储器

利用半导体、磁、光等介质制成存储数据的电子设备。 可分为 4 个层次

  • 片上缓存 处理器核心内的缓存 一般为 SRAM 结构 16k-512k 可分为一级、二级
  • 片外缓存 处理器核心外的缓存，需要经过交换互联开关访问，一般是 SRAM 构成。 256k-4m
  • 主存（内存） 通常采用 DRAM 结构。以独立的部件、芯片存在。依赖不断充电维持其中的数据。x00m-x0g
  • 外存 可以是磁带、磁盘、光盘，访问速度慢，掉电后能够保持其数据。xg-xt。

操作系统

操作系统是配置在计算机硬件上的第一层软件。它向下管理裸机及其文件，向上为其他系统软件提供支持。 操作系统的组成

  • 内核
    • 进程管理（任务管理）
    • 存储管理
    • 文件管理
    • 设备管理
  • 附加的配套软件 操作系统的作用
  • 管理计算机中运行的程序和分配各种软硬件资源。
    • 资源管理主要包括处理器管理、存储管理、文件管理、i/o 设备管理
  • 为用户提供友善的人机界面。
  • 为应用程序的开发和运行提供一个高效率的平台。 操作系统的特征
  • 并发性
    • 在一个 cpu 上多次切换程序或多个 cpu 上分别运行一个程序
  • 共享性
    • 同时共享
    • 互斥共享
  • 虚拟性 指操作系统中的一种管理技术，它是把物理上的一个实体变成逻辑上的多个对应物，或把物理上的多个实体变友逻辑上的一个对应物的技术。
  • 不确定性 系统中的进程何时执行、何时暂停，以什么样的速度向前推进、进程总共要花多少时间执行才能完成，这些都是不可预知的。 操作系统的分类：
  • 批处理操作系统
    • 单道批处理
    • 多道批处理
  • 分时操作系统
    • 多路性
    • 独立性
    • 交互性
    • 及时性
  • 实时操作系统
    • 对交互能力要求不高，对可靠性有保障。
  • 网络操作系统
  • 分布式操作系统 系统中的计算机无主、次这分。任意 2 台计算机可以通过通信交换信息。
  • 微型计算机操作系统
  • 嵌入式操作系统
    • 运行在嵌入式智能设备环境中。
    • 微型化

系统工程

是一种组织管理技术。

  • 把研究的对象或工程管理问题看作一个由很多相互联系、相互制约的组成部分构成的总体。
  • 运用运筹学的理论和方法以及电子计算机技术，对构成系统的各组成部分进行分析、预测和评价
  • 最后进行综合，使该系统达到最优。 系统工程是运用系统方法，对系统进行规划、研究、设计、制造、试验和使用组织管理技术。 系统之系统（System of System SoS）适用于其系统元素本身是系统的情况。 系统工程是从整体出发，合理开发、设计、实施和运用系统科学的工程技术。它根据总体协调的需要，综合应用自然科学和社会科学中有关的思想、理论和方法，利用电子计算机作为工具，对系统的结构、要素，信息和反馈等进行分析，以达到最优规划、最估设计、最优管理和最优控制的目的。 系统工程专注于：
  • 定义客户需求与所要求的功能。将需求文件化。
  • 再考试完整问题，即运行、成本、进度、性能、培训、保障、实验、制造、和退出问题，并进行设计综合和系统确认。 系统工程方法的主要步骤：
  • 对系统提出要求
  • 根据要求设计系统，评价设计方案。
  • 修改要求，再设计。

系统工程的生命周期

定义系统工程的生命周期的目的是以有序而且高效的方式建立一个满足利益攸关者需求的框架。一般通过定义生命周期阶段，并使用一些决策来确定是否处于就绪状态，以便从一个阶段进行下一个阶段来实现这一目的。跳过某些阶段和省去一些决策可能会大幅度增加风险，减少系统工程的投入程度也可能对技术开始造成不利影响。 生命周期阶段

  • 探索性研究阶段 目的是识别利益攸关者的需求，探索创意和技术。
  • 概念阶段 搜索可行概念，提出有望实现的解决方案
  • 开发阶段 旷日持久是细化系统需求，创建解决方案的描述，构建系统，验证并确认系统。
  • 生产阶段 系统被生产或制造的阶段
  • 使用阶段 是系统在预期环境中运行以交付预期服务的阶段
  • 保障阶段 为系统提供服务，使之能持续运行的阶段
  • 退役阶段 系统及其相关服务从运行中移除的阶段 生命周期方法
  • 计划驱动方法

信息系统概述

信息系统是由计算机硬件、网络和通信设备、计算机软件、信息资源、信息用户和规章制度组成的以处理信息流为目的的人机一体化系统。 5 个基本功能：

  • 输入功能
  • 存储功能
  • 处理功能
  • 输出功能
  • 控制功能 人类社会的发展速度将取决于人们对信息的利用水平。 信息系统是以复读机为基础的人机交互系统。 “以计算机为基础”要求系统设计者必须具备计算机及其在信息处理中的应用知识。 “人机交互”要求系统设计者还需要了解人作为系统组成部分的能力以及从作为信息使用者的各种行为。

信息系统的发展

数据处理的发展涉及技术的进步、应用拓展、计划和控制策略的变化以及用户的状况。

    • 一般发生在一个组织的财务部门
  • 传播阶段
    • 数据处理能力得到迅速发展
    • 出现许多新问题（如数据冗余，数据不一致、难以共享等）
    • 计算机使用效率不高
  • 控制阶段
    • 成立了一个领导小组

      • 正向推理或前向推理 数据驱动策略 适用于目标解空间很大的问题
      • 反向推理或逆向推理 适用于目标解空间较小的问题
      • 双向推理或混合推理

    • 对知识编辑或求精

      • 专家系统与用户的接口
      • 专家系统与领域专家户的接口
    • 求解步骤
      • 根据用户的问题在知识库中搜索有相关知识
      • 根据有关的知识和系统的控制策略形成解决问题的途径，即知识操作算子序列。（这时构成一个假设集合）
      • 对解决问题的假设集合进行排序。选了某些准则下的最优假设方案。
      • 根据挑选的解决方案去求解具体问题
      • 如果该方案不能真正解决问题，则追溯到假设方案序列中的下一个，重复求解问题。
      • 重复上述步骤，直到解决问题或宣告“本系统对问题无解”为止。

第 4 章 信息安全技术基础知识

信息安全基础知识

信息安全的 5 个要素：

  • 机密性 确保信息不暴露给未授权的实体或进程
  • 完整性 只有得到允许的人才能修改数据，并且能够判别出是否已被篡改
  • 可用性 得到授权的实体在需要时可访问数据。
  • 可控性 可以控制授权范围内的信息流向及行为方式
  • 可审查性 对出现的信息安全问题提供调查的依据和手段 信息安全的范围：
  • 设备安全
    • 它是首要问题
    • 设备的稳定性 在一定时间内不出故障的概率
    • 设备的可靠性 在一定时间内正常执行任务的概率
    • 设备的可用性 可以正常使用的概率
  • 数据安全 即采取措施确保数据免受未授权的泄露、篡改、毁坏
    • 数据的秘密性 不受未授权者知晓的属性
    • 数据的完整性 数据是正确的、真实的、未被篡改的、完整无缺的属性
    • 数据的可用性 数据可随时正常使用的属性
  • 内容安全 是信息在政治、法律、道德层次上的要求
    • 信息内容在政治上是健康的
    • 信息内容符合国家的法律法规
    • 信息内容符合中华民族优良的道德规范

对称密钥的分配与管理

密钥分配一般要解决 2 个问题：

  • 引进自动分配密钥机制，以提高系统的效率。
  • 尽可能减少系统中驻留的密钥量。 控制密钥的安全性：
  • 密钥标签
  • 控制失量 4 种密钥的分配方式
  • 经过 A 选取的密钥通过物理手段发送给另一方 B
  • 由第三方选取密钥，再通过物理手段分别发送给 A、B
  • AB 事先已经有一个密钥，其中一方选择新密钥后，用已有密钥加密新密钥后发给另一方。
  • 三方 ABC 各有一个保密信道。C 选取密钥后，分别通过 AB 各自的保密信道发送。 no.4 方法最长用。 会话密钥更换得越频繁，系统的安全性就越高。越频繁更换会话密钥，网络负担越重。需要平衡二者。

公钥加密体制的密钥管理

  • 公开发布
    • 用户将自己的公钥发给其他用户。
    • 缺点：任何人都可以发布伪造的别人的公钥。
  • 公用目录表
    • 用一个公用的公钥动态目录表，由某个可信的实体或组织去建立、维护公钥的分布。
    • 每个用户与管理者通信，注册、更新自己的公钥。
    • 缺点：若公用目录表被攻击，则有可能被假冒。
  • 公钥管理机构
    • 用公钥管理机构来为各用户建立、维护动态的公钥目录。
    • 每个用户都可靠地知道管理机构的公钥，但是只有管理机构知道相应的私钥。
    • 缺点：每个用户想和他人联系时须求助于管理机构。所以很容易使管理机构成为系统的瓶颈。
    • 缺点：若公用目录表被攻击，则有可能被假冒。
  • 公钥证书
    • 由证书管理机构 CA（Certificate Authority）为用户建立的，其中的数据项有与该用户的密钥相匹配的公钥、用户身份、时间戳等。
    • 用户将自己的公钥发给另一个用户，则接收方使用 ca 的公钥对证书验证。
    • 这样通过证书交换用户之间的公钥而无须再与公钥管理机构联系。

访问控制及数字签名技术

访问控制技术

访问控制的基本模型 访问控制是指文体依据某些控制策略或权限对客体本身或是其资源进行的不同授权访问。 访问控制的 3 个要素：

  • 主体 对其他实体施加动作的主动实体 简记为 S，有时必称为用户 User，记为 U
  • 客体 接受其他实体访问的被动实体 Object 简记为 O
  • 控制策略 是主体对客体的行为集和约束条件集 简记为 KS 访问控制包括：
  • 认证
  • 控制策略实现
  • 审计 访问控制的实现技术 建立访问控制模型和实现访问控制都是抽象和复杂的行为。
  • 要保证授权用户使用的权限与其拥有的权限对应。
  • 制止非授权用户和非授权行为
  • 保证敏感信息的交叉感染。 具体实现方式
  • 访问控制矩阵 Access Control Matrix ACM
    • 通过矩阵形式表示访问控制规则和授权用户权限的方法。
    • 对每个主体而言，都拥有哪些客体的哪些访问权限。
    • 对客体而言，有哪些主体可以对它实施访问。
    • 主体是行索引，客体是列索引。矩阵中每个元素表示一组访问方式。
    • 易于理解，实现起来有一定难度。会呈几何数增长。会有好多空值。
  • 访问控制表 Access Control Lists ACLs
    • 是目前最流行、使用最多的访问控制实现技术
    • 每个客体有一个访问控制表，是系统中每个有权访问这个客体的主体的信息。
  • 能力表 Capabilities
    • 每个主体有一个能力表（Cap-ability Lists），表示主体对系统中每个客体的访问权限信息。
  • 授权关系表
    • 是某一个主体对应于某一个客体的访问权限信息
    • 若授权关系表按主体排序，则可以得到能力表的效率；若授权关系表按客体排序，则可以得到访问控制表的效率，

数字签名

向通信双方提供服务， 可以做到：

  • B 可以验证消息 P 确实来源于 A
  • A 以后不能否认发送过 P
  • B 不能编造或改变消息 P 数字签名的条件
  • 签名是可信的。
  • 签名不可伪造
  • 签名不可重用
  • 签名的文件不可改变
  • 签名是不可抵赖的。 对称密钥签名 公开密钥签名
  • 数字签名协议经常和单向 hash 函数一起和，即只对文件的 hash 值签名。
  • 优点：
    • 签名和文件可以分开保存
    • 接收者对文件和签名的存储量要求低。
    • 档案系统可用这些协议来验证文件的存在而不需保存它们的内容
    • 用户将文件的 hash 值传给数据库

    • 大量地申请系统资源
    • 不断地向服务程序发请求
    • 可攻击的资源

  • 破坏或更改配置信息

  • 利用服务程序中的处理错误使服务失效 分布或拒绝服务攻击 DDoS 相比 dos，可以使用更多进攻资源，隐蔽性好。 被 DDoS 攻击的表现：
  • 有大量的等待的 tcp 连接
  • 大量到达的数据分组（）并不是网站服务连接的一部分。往往指向机器的任意接口。

  • 制造高流量的无用数据造成网络拥塞，使受害主机无法正常与外界通信
  • 利用受害主机提供的服务和传输协议上的缺陷，反复发出服务请求，使受害主机无法及时处理所有正常请求。

系统结构图 Structure Chart SC

又称模块结构图。它是软件概要设计阶段的工具，反映系统的功能实现和模块之间的联系与通信，包括名模块之间的层次结构，即反映了系统的总体结构。 详细设计的主要任务是设计每个模块的实现算法、所需的局部数据结构。目标：

  • 实现模块功能的算法要逻辑上正确和算法描述要简明易懂。
  • 若要更改概要设计，则必须经过项目经理的同意。 设计的基本步骤：
  1. 分析并确定输入、输出数据的逻辑结构
  2. 找出输入数据的结构和输出数据结构中对应关系的数据单元
  3. 按一定的规则由输入、输出的数据结构导出程序结构。
  4. 列出基本操作与条件，并把它们分配到程序结构图的适当位置。
  5. 用伪码写出程序。 详细设计的表示工具：
  • 图形工具

    • 程序流程图（也叫程序框图）

    • NS流程图（也叫盒图）
    • Problem Analysis Diagram PAD图
      • 可取代程序流程图。
      • 特点：

        • pad所描绘的程序结构十分清晰
        • 用pad图表现程序的逻辑易读、易懂和易记
        • 容易将PAD图转换成高级语言源程序自动完成

        • 支持自顶向下方法的使用。

软件测试

使用人工或自动的手段来运行或测定某个软件系统的过程，其目的在于检验它是否满足规定的需求或弄清预期结果与实际结果之间的差别。 重要的是软件测试要贯穿在整个软件开发的过程中，保证整个软件开发的过程是高质量的。

测试方法

按执行状态分为静态测试、动态测试 按实现算法细节和系统内部结构分为黑盒测试、白盒测试、灰盒测试。 按程序执行方式分为人工测试、自动化测试

  • 静态测试
    • 以源程序做结构分析和流程图分析
    • ts就是使用这种测试
  • 动态测试
    • 构造测试实例
    • 执行程序
    • 分析结果
  • 黑盒测试
    • 不考虑内部结构，根据需求规格测试实例。检查功能是否正确运行。
    • 主要对软件界面和软件功能进行测试
    • 必须加以量化才能够保证软件的质量。
  • 白盒测试
    • 从程序结构方面设计测试用例。用于检查各逻辑是否合理。
    • 对应的模块独立路径是否正常。
    • 常用方法
      • 控制流分析
      • 数据流分析
      • 路径分析
      • 程序变异
  • 灰盒测试
  • 人工测试
  • 自动化测试

软件质量保证 Software Quality Assurance SQA 是建立一套有计划，有系统的方法，来向管理层保证拟定写的标准、步骤、实践和方法能够被所有项目所采用。 质量保证的主要目标：

  • 事前预防工作，例如，着重于缺陷预防而不是缺陷检查。
  • 尽量在刚刚引入缺陷时将其捕获（不让其扩散到下一个阶段）
  • 作用于过程 （不是最终产品）
  • 贯穿于所有的活动之中（不是只集中于一点） SQA保证质量的方法：
  • 选定开发方法被采用
  • 选定的标准和规程得到采用和遵循
  • 进行独立的审查
  • 偏离标准和规程的问题得到及时的反映和处理
  • 项目定义的每个软件任务得到实际的执行 SQA主要任务：
  • 审计与评审
  • 报告
  • 处理不符合问题 软件质量认证

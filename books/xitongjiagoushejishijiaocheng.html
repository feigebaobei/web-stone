系统架构概述

  • 系统架构师是项目的总设计师，他需要掌控整体、洞悉局部瓶颈。依据具体的业务场景给出解决方案的总体设计人员。
  • 他要确认和主体系统需求，给出开发规范，搭建系统实现的核心架构，并澄清技术细节、扫清主要难点的技术人员
  • 掌握技术团队的能力需要，给出项目管理方法，采用合适生命周期模型，具备以自身为核心形成团队的能力。
  • 在项目进度计划和经费分配等方面开展评估，以预防项目风险。

系统架构设计的目的对需要开发的系统进行一系列相关的抽象，用于指导系统各个方面的设计与实现。 架构设计的作用主要包括：

  • 解决相对复杂的需求分析问题
  • 解决非功能属性在系统占据重要位置的设计问题
  • 解决生命周期行、扩展性需求高的系统整体结构问题
  • 解决系统基于组件需要的集成问题
  • 解决业务流程再造难的问题 软件架构（也可称为体系结构）是用来刻画软件系统整体抽象结构的一种手段。

微核架构

又称为插件架构。指软件的内核相对较小，主要功能和业务逻辑都通过插件实现。 插件是相互独立的。插件之间的通信应该减少到最低，避免出现互相依赖的问题。

3 种实现模式

  • RESTful API 模式 服务通过 api 提供，云服务就属于这一类。
  • RESTful 应用模式 通过传统的网络协议或者应用协议提供，背后通常是一个多功能的应用程序。
  • 集中消息模式 采用消息代码可以实现消息队列、负载均衡、统一日志和异常处理。缺点是会出现单点失败，消息代理可能要做成集群。

云架构

主要解决扩展性和并发的问题。是最容易扩展的架构。

处理单元 实现业务逻辑 虚拟中间件 负责通信、保持会话控制（sesssion）、数据复制、分布式处理和处理单元的部署 消息中间件 管理用户请求和会话控制，当一个请求进来以后，它决定分配给哪个处理单元。 数据中间件 用于数据同步。使每个处理单元得到同样数据。 处理中间件 可选，当一个请求涉及不同类型的处理单元，则该中间件负责协调处理单元。 部署中间件 负责处理单元的启动和关闭，监控负载和响应时间 ，当负载增加时，则启动新处理单元，当负责减小时，则关闭处理单元。

系统架构的常用建模方法

软件架构建模是软件架构的表示方法。

  • 结构模型 最直观，最普遍。以架构的构件、连接件和其他概念来刻画结构。
  • 框架模型 与结构模型类似，不太侧重描述结构的细节，更侧重整体的结构。建立只针对问题的结构。
  • 动态模型 对结构或框架模型的补充。
  • 过程模型 研究构造系统的步骤和过程，其结构是遵循过程脚本的结果。

软件架构可从不同角度来描述用户所关心架构的特征。

  • 逻辑视角
  • 过程视角
  • 物理视角
  • 开发视角
  • 场景视角

不同的架构风格具有不同的优缺点和应用场景 软件架构的应用场景。

  • 管道-过滤器风格 适用于系统分成若干独立的步骤
  • 主程序、子系统和面向对象的架构风格 用于对组件内部进行设计
  • 虚拟机风格 常用于构造解释器或专家系统
  • cs/bs 风格 适用于数据和处理分布在一定范围，通过网络连接构成系统
  • 平台、插件风格 用于具有插件扩展功能的应用程序
  • mvc 风格 广泛应用于用户交互程序的设计
  • soa 风格 应用在企业集成等方面
  • c2 风格 适用于 gui 软件开发，用于构建灵活和可扩展的应用系统。 在现代大型软件中，常使用多种架构风格。并保证软件系统的可靠性、可扩展性、可维护性。

从工程师到系统架构设计师的演化

工程师和系统架构设计师的区别主要体现在技术、组织、个人成长上。 业务领域 > 抽象建模 架构设计师

  • 是技术团队的对外接口人，也是外部团队技术问题的终结者。
  • 架构工作就是权衡。 工程师
  • 必须熟练掌握详细的设计方法、编程语言、工具和环境。
  • 经常是完美主义

成为系统架构设计师的关键是要培养自己在判断力、执行力和创新力。 判断力 能够准确判断系统的复杂度在哪里，能准确看出系统的弱点。 执行力 能够使用合适的方案解决复杂度问题。 创新力 能够创建新的解决方案解决复杂问题。

工程师阶段

1-3 年 在别人的指导下完成开发。 积累基础知识。包括编程语言、基本数据结构、开发环境、操作系统、数据库以及相关软件开发流程。

高级工程师阶段

3-5 年 独立完成开发 数据库表设计的 3 个范式、面向对象的设计模式、solid 设计原则、缓存设计理论（缓存穿透、缓存雪崩、缓存热点）

技术专家阶段

4-8 年 可以解决此领域内的问题。 根据需要修改、扩展和优化架构。在高级工程师的基础上扩展技术宽度。

系统架构师（初级）

5-8 年 独立完成一个系统的架构设计。可以是 0-1 设计一个新系统，也可以是 1.0-2.0 重构一个系统。 基于完善的架构设计方法论的指导来进行架构设计。技术专家是基于经验进行架构设计的。 在技术专家的基础上形成自己的架构设计方法论。

系统架构师（中级）

8-10 年 能够完成复杂系统的架构设计。包含高性能、高可用、可扩展、海量存储等复杂系统。 基于初级系统架构师加深技术深度和技术理论的积累。

系统架构师（高级）

10+年 创造新的架构模式。如分布式存储架构、分布式计算 MapReduce 架构和列式存储架构等开创了大数据时代；在虚拟机很成熟的背景下，docker 创建了容器化的技术潮流。 创建新的架构模式，开创新的技术潮流。 基于中级系统架构师再创造性。

存储器

利用半导体、磁、光等介质制成存储数据的电子设备。 可分为4个层次

  • 片上缓存 处理器核心内的缓存 一般为SRAM结构 16k-512k 可分为一级、二级
  • 片外缓存 处理器核心外的缓存，需要经过交换互联开关访问，一般是SRAM构成。 256k-4m
  • 主存（内存） 通常采用DRAM结构。以独立的部件、芯片存在。依赖不断充电维持其中的数据。x00m-x0g
  • 外存 可以是磁带、磁盘、光盘，访问速度慢，掉电后能够保持其数据。xg-xt。

计算机软件概述

操作系统

操作系统是配置在计算机硬件上的第一层软件。它向下管理裸机及其文件，向上为其他系统软件提供支持。 操作系统的组成

  • 内核
    • 进程管理（任务管理）
    • 存储管理
    • 文件管理
    • 设备管理
  • 附加的配套软件 操作系统的作用
  • 管理计算机中运行的程序和分配各种软硬件资源。
    • 资源管理主要包括处理器管理、存储管理、文件管理、i/o设备管理
  • 为用户提供友善的人机界面。
  • 为应用程序的开发和运行提供一个高效率的平台。 操作系统的特征
  • 并发性
    • 在一个cpu上多次切换程序或多个cpu上分别运行一个程序
  • 共享性
    • 同时共享
    • 互斥共享
  • 虚拟性 指操作系统中的一种管理技术，它是把物理上的一个实体变成逻辑上的多个对应物，或把物理上的多个实体变友逻辑上的一个对应物的技术。
  • 不确定性 系统中的进程何时执行、何时暂停，以什么样的速度向前推进、进程总共要花多少时间执行才能完成，这些都是不可预知的。 操作系统的分类：
  • 批处理操作系统
    • 单道批处理
    • 多道批处理
  • 分时操作系统
    • 多路性
    • 独立性
    • 交互性
    • 及时性
  • 实时操作系统
    • 对交互能力要求不高，对可靠性有保障。
  • 网络操作系统
  • 分布式操作系统 系统中的计算机无主、次这分。任意2台计算机可以通过通信交换信息。
  • 微型计算机操作系统
  • 嵌入式操作系统
    • 运行在嵌入式智能设备环境中。
    • 微型化
    • 可定制
    • 实时性
    • 可靠性
    • 易移植性

软件构件

构件也叫组件。是一个自包容、可复用的程序集。 软件构件的组装模型的优点：

  • 构件的自包含性让系统的扩展变得更加容易。
  • 设计良好的构件更容易被重用，降低软件开发成本。
  • 构件的粒度较整个系统更小。因此安排开发任务更加灵活。可以把项目会成若干组可以独立开发的组件。 软件构件的组装模型的缺点：
  • 对构件设计需要经验丰富的架构设计师，设计不良的构件难以实现构件的优点，反而会降低组件模型的重用度。
  • 需要在重用度与性能之间平衡。
  • 要求程序员能够熟练地使用构件组装应用程序。
  • 第三方的构件库会影响软件的质量。 商用构件的标准规范

系统工程

是一种组织管理技术。

  • 把研究的对象或工程管理问题看作一个由很多相互联系、相互制约的组成部分构成的总体。
  • 运用运筹学的理论和方法以及电子计算机技术，对构成系统的各组成部分进行分析、预测和评价
  • 最后进行综合，使该系统达到最优。 系统工程是运用系统方法，对系统进行规划、研究、设计、制造、试验和使用组织管理技术。 系统之系统（System of System SoS）适用于其系统元素本身是系统的情况。 系统工程是从整体出发，合理开发、设计、实施和运用系统科学的工程技术。它根据总体协调的需要，综合应用自然科学和社会科学中有关的思想、理论和方法，利用电子计算机作为工具，对系统的结构、要素，信息和反馈等进行分析，以达到最优规划、最估设计、最优管理和最优控制的目的。 系统工程专注于：
  • 定义客户需求与所要求的功能。将需求文件化。
  • 再考试完整问题，即运行、成本、进度、性能、培训、保障、实验、制造、和退出问题，并进行设计综合和系统确认。 系统工程方法的主要步骤：
  • 对系统提出要求
  • 根据要求设计系统，评价设计方案。
  • 修改要求，再设计。

系统工程的生命周期

定义系统工程的生命周期的目的是以有序而且高效的方式建立一个满足利益攸关者需求的框架。一般通过定义生命周期阶段，并使用一些决策来确定是否处于就绪状态，以便从一个阶段进行下一个阶段来实现这一目的。跳过某些阶段和省去一些决策可能会大幅度增加风险，减少系统工程的投入程度也可能对技术开始造成不利影响。 生命周期阶段

  • 探索性研究阶段 目的是识别利益攸关者的需求，探索创意和技术。
  • 概念阶段 搜索可行概念，提出有望实现的解决方案
  • 开发阶段 旷日持久是细化系统需求，创建解决方案的描述，构建系统，验证并确认系统。
  • 生产阶段 系统被生产或制造的阶段
  • 使用阶段 是系统在预期环境中运行以交付预期服务的阶段
  • 保障阶段 为系统提供服务，使之能持续运行的阶段
  • 退役阶段 系统及其相关服务从运行中移除的阶段 生命周期方法
  • 计划驱动方法
  • 渐进迭代式开发
    • 允许为项目提供一个初始能力，随之提供连续交付以达到期望的系统。
    • 目标在于快速产生价值并提供快速响应能力。
  • 精益开发
    • 是一种整体性的范式，聚集于向客户交付最大价值并使浪费活动最小化
  • 敏捷开发
    • 目标在于灵活性
    • 原则
      • 最高的优先级是通过尽早地和持续地交付有价值的软件来满足客户
      • 欢迎需求变更。即使在项目开发后期。
      • 不断交付可用的软件。周期越知越好
      • 在项目中开发人员与业务人员每天在一起工作，业务人员始终参与到开发工作中。
      • 开发团队内，多以面对面式交谈。
      • 工作软件是进度的主要度量
      • 对技术的精益求精以及对设计的不断完善将提升敏捷性
      • 简单性是精髓
      • 最佳的架构、需求和设计出自于自组织的团队
      • 团队要定期反省如何做到更加高效，并相应地调整团队的行为。

性能评估

是为了一个目的，按照一定的步骤，选用一定的度量项目，通过建模和实验，对一个系统的性能进行各项检测，对测试结果做出解释，并形成一份文档的技术。 性能评估的一个目的是为性能的优化提供参考。

  • 基准测试程序 把应用程序中使用最多、最频繁的那部分核心程序作为评价计算机性能的标准程序。
